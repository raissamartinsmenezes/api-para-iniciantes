    *Introdução

    Para nos ancorar, vamos começar com algo que é físico: o servidor (the server). Um servidor nada mais é do que um grande computador, tem todas as mesmas partes que um laptop desktop que você usa para o trabalho, é apenas mais rápido e mais poderoso. Normalmente, os servidores não têm um monitor, teclado ou mouse, o que os torna inacessíveis.

    Da mesma forma que um programa como a Paciência espera por você clicar
    em um cartão para fazer algo, um servidor web executa um programa que aguarda uma pessoa pedir por um página da web. Não há nada de mágico ou espetacular nisso. Um desenvolvedor de software grava um programa, copia-o para um servidor que o executa continuamente.

    O que é uma API?

    Você pode olhar para um formulário em um site e saber que a caixinha com a frase "Nome" significa que você deve digitar a palavra que você usa para se identificar informalmente. No entanto, o que acontece quando você enfrenta uma tarefa muito demorada, como copiar as informações de contato de mil clientes de um site para outro? Você adoraria delegar esse trabalho a um computador para que ele possa ser feito com rapidez e precisão. Infelizmente, as características que tornam os sites ideais para o uso dos seres humanos, dificultam o uso para os computadores. A solução é uma API (Application Programming Interface), a ferramenta que faz os dados de um site digerível para um computador. Através dela, um computador pode ver e editar dados, assim como uma pessoa pode carregar páginas e enviar formulários.

    Como uma API é usada?

    Quando dois sistemas (sites, desktops, smartphones) se conectam a uma API, dizemos que eles são integrados (integrated). Em uma integração, você tem dois lados, cada um com um nome especial. Um lado nós já falamos sobre: o servidor. O outro lado é o cliente (client). Este conhece quais dados estão disponíveis por meio da API e pode manipulá-los, geralmente através de uma solicitação (request) do usuário.

    *Protocolo

    Um protocolo de computador é um conjunto aceito de regras que governam como dois computadores podem falar um com o outro. Comparado ao nosso padrões, no entanto, um protocolo de computador é extremamente rígido. Pense por um momento das duas frases "Minha cor favorita é azul" e "Azul é minha cor favorita." As pessoas são capazes de quebrar cada frase e ver que eles significam a mesma coisa, apesar das palavras estarem em diferentes ordens. Infelizmente, os computadores não são tão inteligentes. Para dois computadores se comunicarem efetivamente, o servidor precisa saber exatamente como o cliente organizará suas mensagens. Você pode pensar nisso como um pessoa pedindo
    um endereço de correspondência.

    O protocolo da Web

    Existe um protocolo para praticamente tudo; cada um adaptado para fazer trabalhos diferentes. Você já deve ter ouvido falar de alguns: Bluetooth para dispositivos de conexão e POP ou IMAP para buscar e-mails. Na web, o protocolo principal é o Protocolo de Transferência de Hipertexto, mais conhecido por sua sigla, HTTP (Hyper-Text Transfer Protocol). Quando você digita um endereço como http://exemplo.com em um navegador da web, o "http" informa ao navegador para que ele use as regras do HTTP ao falar com o servidor. Com a onipresença do HTTP na web, muitas empresas optam por adotá-lo como o protocolo subjacente às suas APIs. Um benefício de usar um protocolo familiar é que diminui a curva de aprendizado para os desenvolvedores, que incentiva o uso da API. Outro benefício é que o HTTP tem vários recursos úteis na criação de uma boa API, como veremos mais adiante. Certo, agora, vamos enfrentar a água e dar uma olhada em como o HTTP funciona!

    Solicitação HTTP

    A comunicação em HTTP é centrada em torno de um conceito chamado Ciclo de Solicitação-Resposta. O cliente envia ao servidor uma solicitação para fazer alguma coisa. O servidor, por sua vez, envia ao cliente uma resposta dizendo se o servidor, pode ou não, fazer o que o cliente está pedindo.

    Para fazer uma solicitação válida, o cliente precisa incluir quatro coisas:
    1 URL (Localizador Uniforme de Recursos - Uniform Resource Locator)
    2 Método (method)
    3 Lista de Cabeçalhos (headers)
    4 Corpo (body)
    Isso pode soar como um monte de detalhes apenas para passar uma mensagem, mas lembre-se, os computadores têm que ser muito específicos para se comunicar com um outro.

    Resposta HTTP

    Depois que o servidor recebe uma solicitação do cliente, ele tenta atender o pedido e enviar ao cliente uma resposta (response). Respostas HTTP têm uma estrutura muito semelhante às solicitações. A principal diferença é que, em vez de um método e uma URL, a resposta inclui um código de status (status code). Além disso, os cabeçalhos de resposta e o corpo seguem o mesmo formato das solicitações.

    Como as APIs são construídas?

    Até agora, você pode ver que o HTTP suporta uma ampla gama de permutações para ajudar o cliente e o servidor a falar. Então, como isso nos ajuda com APIs? A flexibilidade do HTTP significa que as APIs construídas nele podem fornecer aos clientes um grande potencial de negócios. Um simples ajuste no método de solicitação é a diferença entre dizer ao servidor para criar um novo pedido ou cancelar um existente. É simples transformar o resultado desejado em uma instrução que o servidor possa entender. Essa versatilidade no protocolo HTTP se estende a outras partes de uma solicitação também. Algumas APIs exigem um cabeçalho específico, enquanto outras exigem informações específicas dentro do corpo da solicitação. Ser capaz de usar APIs depende de saber como fazer a solicitação HTTP correta para obter o resultado desejado.

    Pedindo sua pizza!

    Aqui está um exemplo para ajudar a ilustrar esses métodos. Digamos que há uma pizzaria com uma API que você pode usar para fazer pedidos. Você faz um pedido fazendo uma solicitação PUT para o servidor do restaurante com os detalhes do seu pedido, solicitando que ele crie sua pizza. Assim que você enviar a solicitação, no entanto, você perceberá que escolheu a crosta de estilo errada, portanto, você faz uma solicitação PUT para alterá-la. Enquanto aguarda o seu pedido, você faz um monte de solicitações GET para verificar o status. Após uma hora de espera, você decide que já teve o suficiente e faz uma solicitação DELETE para cancelar seu pedido. Lembrando que é no corpo (body) onde os detalhes do pedido vão.

    Instruções:
    1 Envie uma solicitação GET geralmente sem nenhum dado do corpo.
    2 Envie uma solicitação POST e digite seu tipo favorito de pizza no campo do corpo.
    3 Envie uma solicitação PUT e digite um novo ingrediente para adicionar ao seu pizza no campo do corpo.
    4 Envie uma solicitação DELETE sem nenhum dado do corpo.
